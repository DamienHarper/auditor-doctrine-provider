# Upgrading to 2.0

> **Migrate from auditor-doctrine-provider 1.x to 2.0**

This guide covers upgrading from auditor-doctrine-provider 1.x to 2.0.

## Overview

auditor-doctrine-provider 2.0 introduces **schema version 2**: a revised audit table format
with a unified `blame` JSON column, a ULID-based `transaction_id`, and a richer `diffs`
envelope. Existing data is fully preserved ŌĆö migration is additive and reversible in the
DDL phase.

## ŌÜĀ’ĖÅ Who is affected?

| Scenario | Impact |
|----------|--------|
| Using `damienharper/auditor-bundle` 8.x | Ō£ģ Bundle handles wiring automatically |
| Using `auditor-doctrine-provider` standalone | ŌÜĀ’ĖÅ Must migrate audit tables manually |
| Custom providers calling `persist(LifecycleEvent $event)` | ŌÜĀ’ĖÅ Payload keys have changed |
| Code reading `$entry->transactionHash` | ŌÜĀ’ĖÅ Property removed ŌĆö use `$entry->transactionId` |

## ­¤ÜĆ Upgrade Steps

### 1’ĖÅŌāŻ Update the package

```bash
composer require damienharper/auditor-doctrine-provider:^2.0 damienharper/auditor:^5.0
```

### 2’ĖÅŌāŻ Migrate existing audit tables

> [!CAUTION]
> `audit:schema:update` refuses to run while any audit table still has the legacy v1 schema.
> Always run `audit:schema:migrate` first.

```bash
# Preview the changes (no writes)
bin/console audit:schema:migrate --dump-sql

# Run the full migration: DDL + transaction hash conversion + diffs conversion
bin/console audit:schema:migrate --force --convert-all
```

> [!TIP]
> The individual flags `--convert-transaction-hash` and `--convert-diffs` are available for
> step-by-step migration on large datasets, and can be run in any order.

After migration:

```bash
# Apply any remaining schema changes (new indexes, etc.)
bin/console audit:schema:update --force
```

### 3’ĖÅŌāŻ Update code reading `$entry->transactionHash`

```php
// Before (1.x)
$entry->transactionHash;    // SHA-1 VARCHAR(40) string

// After (2.0)
$entry->transactionId;      // ULID CHAR(26) string
```

## ­¤Śä’ĖÅ Schema Changes

| Column (1.x) | Column (2.0) | Notes |
|---|---|---|
| `transaction_hash VARCHAR(40)` | `transaction_id CHAR(26)` | SHA-1 replaced by ULID |
| `blame_user VARCHAR(255)` | *(removed)* | Merged into `blame` JSON |
| `blame_user_fqdn VARCHAR(255)` | *(removed)* | Merged into `blame` JSON |
| `blame_user_firewall VARCHAR(100)` | *(removed)* | Merged into `blame` JSON |
| `ip VARCHAR(45)` | *(removed)* | Merged into `blame` JSON |
| *(new)* | `blame JSON NULL` | `{"username","user_fqdn","user_firewall","ip"}` |
| *(new)* | `schema_version SMALLINT DEFAULT 1` | `1` = legacy rows, `2` = migrated rows |

Two new composite indexes are also added:

```sql
CREATE INDEX idx_object_id_type   ON audit_table (object_id, type);
CREATE INDEX idx_blame_id_created ON audit_table (blame_id, created_at);
```

## ­¤öä Breaking Changes

### `Entry::$transactionHash` removed

Property removed with no deprecated alias. Replace all reads with `$entry->transactionId`.

### `LifecycleEvent` payload keys changed

Custom providers must update any code reading the old flat keys from `$event->getPayload()`:

| Key (1.x) | Key (2.0) | Notes |
|---|---|---|
| `transaction_hash` | `transaction_id` | ULID string |
| `blame_user` | `blame['username']` | Nested in `blame` array |
| `blame_user_fqdn` | `blame['user_fqdn']` | Nested in `blame` array |
| `blame_user_firewall` | `blame['user_firewall']` | Nested in `blame` array |
| `ip` | `blame['ip']` | Nested in `blame` array |
| *(new)* | `schema_version` | `2` for all new entries |

### `getDiffs()` return value for schema v2 rows

For entries written with `schema_version >= 2`, `getDiffs()` returns the `changes` sub-array
from the unified diffs envelope. Legacy v1 rows are unaffected.

```php
// schema_version = 1: raw ['field' => ['old' => x, 'new' => y], ...]
// schema_version >= 2: envelope['changes'] ŌĆö same shape
$changes = $entry->getDiffs();
```

### `audit:schema:update` now guards against v1 tables

If any audit table still has the legacy `transaction_hash` column, `audit:schema:update`
will refuse to run and print an error listing the affected tables. Run
`audit:schema:migrate --force` first.

## Ō£© New Features

### `$entry->transactionId` ŌĆö ULID

```php
$entry->transactionId;  // '01JXXXXXXXXXXXXXXXXXXXXXXXXX' (26 chars)
```

### `$entry->blame` ŌĆö full blame context

```php
$entry->blame;  // ['username' => '...', 'user_fqdn' => '...', 'user_firewall' => '...', 'ip' => '...']
```

### `$entry->schemaVersion`

```php
$entry->schemaVersion;  // 1 (legacy) or 2 (current)
```

### `Entry::getDiffSource()` ŌĆö entity metadata

```php
$source = $entry->getDiffSource();
// ['id' => '42', 'class' => 'App\Entity\Post', 'label' => 'My post', 'table' => 'post']
// Returns null for schema_version = 1 entries.
```

### `Entry::getDiffTarget()` ŌĆö association target

```php
$target = $entry->getDiffTarget();
// ['id' => '7', 'class' => 'App\Entity\Tag', 'label' => 'PHP', 'table' => 'tag']
// Returns null for non-association entries or schema_version = 1.
```

### Reader backwards compatibility

The `Reader` automatically adapts queries to use `transaction_id` or `transaction_hash`
depending on which column exists in each table. Mixed environments (some tables migrated,
some not) are fully supported.
